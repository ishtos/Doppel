
## 作業計画（ステアリングファイル）

複数ステップにわたる開発タスクでは、作業開始前にステアリングファイルを作成して計画を管理します。
作業完了後は参照用として保持されますが、新しい作業では新しいディレクトリを作成します。

### 作成基準

以下のいずれかに該当する場合、ステアリングファイルを作成:

複数ファイルにまたがる変更
新規API・機能の追加
データベーススキーマの変更
既存機能の大幅なリファクタリング

単純なバグ修正や1ファイルの小さな変更では不要です。

### ディレクトリ構造

```
.steering/
└── [YYYYMMDD]-[開発タイトル]/
    ├── requirements.md    # 要件定義
    ├── design.md          # 設計書
    └── tasklist.md        # タスクリスト
```

### 各ファイルの役割

**requirements.md**
変更・追加する機能の説明
ユーザーストーリー
受け入れ条件
制約事項

**design.md**
実装アプローチ
変更するコンポーネント・ファイル一覧
データ構造・スキーマの変更
API設計（エンドポイント、リクエスト/レスポンス）
影響範囲の分析

**tasklist.md**
具体的な実装タスク（チェックボックス形式）
各タスクの完了条件
進捗状況の記録

```markdown
# タスクリスト

## フェーズ1: データベース
[x] マイグレーションファイル作成
[x] Prismaスキーマ更新
[ ] シードデータ追加

## フェーズ2: API実装
[ ] エンドポイント作成
[ ] バリデーション実装
[ ] テスト作成
```

### ワークフロー

**計画**: `.steering/[YYYYMMDD]-[タイトル]/` ディレクトリを作成
**要件定義**: `requirements.md` に要件を記載
**設計**: `design.md` に設計を記載
**タスク分解**: `tasklist.md` にタスクを分解
**実装**: タスクを順次実行、完了時にチェック
**完了**: 全タスク完了後、コミット・プッシュ

---

## Git ワークフロー

### コミットメッセージ

```
<type>: <description>

<optional body>
```

Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`

### ブランチ戦略

`main` - 本番環境
`feature/*` - 機能開発
`fix/*` - バグ修正
